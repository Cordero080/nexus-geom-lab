## Task 1.5.3: CSS Specificity & Cascade (25 minutes)

### The Navbar Styling Challenge

**Why this matters:** Your navbar has different styles on different pages, using CSS specificity rules.

**Files to examine:**

- `src/index.css` (find the `.quantum-nav` rules)

### Understanding the Specificity System

**Look at these CSS rules in index.css:**

```css
/* Base rule - applies to all navbars */
.quantum-nav {
  ...;
}

/* More specific - only non-GeomLab pages */
body:not(.geom-lab-page) .quantum-nav {
  ...;
}

/* Most specific - only GeomLab pages */
.geom-lab-page .quantum-nav {
  ... !important;
}
```

### Key Concepts

1. **Specificity hierarchy:**

   - Element selectors (lowest)
   - Class selectors
   - ID selectors
   - Inline styles
   - !important (highest)

2. **How your app uses this:**
   - Base navbar styles for all pages
   - Enhanced styles for most pages
   - Override styles for GeomLab

## Task 1.5.4: Performance & Best Practices (20 minutes)

### CSS Performance in Your App

**Why this matters:** AI often generates styles that work but aren't optimized.

### Performance Patterns in Your Code

**Look for these patterns in your app:**

1. **CSS Transforms over position changes:**

   ```css
   transform: translate3d(${x}px, ${y}px, 0);  /* Good - GPU accelerated */
   /* vs */
   left: ${x}px; top: ${y}px;  /* Bad - causes layout */
   ```

2. **will-change property:**

   ```css
   will-change: transform, opacity; /* Tells browser to optimize */
   ```

3. **pointer-events optimization:**
   ```css
   pointer-events: none; /* Prevents blocking mouse events */
   ```

### Find These in Your Code

**Search your codebase for:**

- `transform: translate3d` (in parallax layers)
- `will-change` (in animated elements)
- `pointer-events: none` (in background layers)

### Why These Matter

- **Transforms:** Use GPU instead of CPU
- **will-change:** Browser pre-optimizes elements
- **pointer-events:** Prevents performance bottlenecks

**‚úÖ Checkpoint:** You should find these patterns in your parallax system and understand why they're used.

## Task 1.5.5: Responsive Design Patterns (20 minutes)

### Viewport Units and Fluid Design

**Your app uses modern responsive techniques:**

**Look for these patterns:**

1. **Viewport units:**

   ```css
   height: 40vh; /* 40% of viewport height */
   width: 100vw; /* 100% of viewport width */
   ```

2. **Clamp and min/max:**

   ```css
   font-size: clamp(16px, 4vw, 24px); /* Fluid typography */
   ```

3. **Flexible layouts:**
   ```css
   display: flex;
   justify-content: space-between;
   ```

### Find in Your Code

**Search for:**

- `vh` and `vw` units (viewport sizing)
- `clamp()` functions (fluid scaling)
- `flex` properties (flexible layouts)

### Why These Matter

- **Viewport units:** Scale with screen size automatically
- **Clamp:** Provides min/max bounds with fluid scaling
- **Flexbox:** Handles layout without media queries

**‚úÖ Checkpoint:** You should understand how your app adapts to different screen sizes.
